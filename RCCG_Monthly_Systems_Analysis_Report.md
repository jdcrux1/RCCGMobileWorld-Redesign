# 2. Site Infrastructure & Architectural Framework

Our front-end architecture is built upon a modular, highly scalable foundation designed to accommodate rapid growth and high traffic volumes during global events. 

### 2.1 Standardized Templating and Consistency
The site ecosystem strictly adheres to a unified design baseline. All primary pages, including our core ministries, sermon libraries, and event schedules, inherit from heavily structured, modular HTML templates (`index.html`, `pages/*.html`). 
- **CSS Architecture:** We leverage scoped, utility-first styling (`Tailwind CSS` alongside optimized vanilla CSS in `style.css` and `pages-shared.css`). This eliminates technical debt commonly associated with monolithic stylesheets, ensuring that UI updates propagate cleanly without inducing regressions.
- **Component Reusability:** Critical shared components—such as the global navigation header, interactive sliders, search overlays, and footers—are managed via dynamic synchronization scripts. This guarantees that any localized change is universally synchronized across the 15+ subpages of the application.

### 2.2 Performance & Responsiveness
Performance profiling remains a key operative metric. Imagery is optimized, and core DOM execution is deferred where possible to guarantee sub-second Time-to-Interactive (TTI) scores across mobile and terrestrial bandwidths.

# 3. Version Control & Collaborative Engineering (Git Lifecycle)

To mitigate operational risk and eliminate single points of failure, the entirety of the site's source code has been securely deposited into a **Public Node Git Repository**. This foundational shift moves us away from localized, fragmented development and aligns the ICT Department with elite industry standards.

### 3.1 Repository Governance
- **Centralized Source of Truth:** The `origin/main` branch acts as the immutable master record of the production environment. 
- **Software Unit Integration:** The Public Git structure grants immediate, asynchronous access to the Software Unit. Engineers can clone the repository, run local instances, and author discrete feature branches.
- **Pull Request (PR) Mandate:** Direct commits to the main branch are strictly prohibited. All modifications—whether structural HTML additions or CSS refinements—must be submitted via Pull Requests. This ensures that every change undergoes rigorous peer review, code quality scanning, and architectural validation before integration.

---

# 4. Continuous Integration & Deployment Pipeline (Vercel)

A modern web presence demands absolute confidence before any code reaches the congregation. To that end, we have fully integrated our Git repository with **Vercel**—a cutting-edge cloud platform structured optimized for immediate deployment and global edge caching.

### 4.1 Staging and Quality Assurance (QA)
1. **Automated Preview Environments:** Every time a Pull Request is generated by the Software Unit, Vercel automatically deploys a carbon-copy, live online testing instance of the site.
2. **Pre-Production Testing:** This isolated link allows the QA Unit, Front-End Architects, and authorized Board members to interact identically with the staged changes in a live browser context—ensuring that the UX flows precisely as intended.
3. **Approval Gateways:** No feature goes live without passing this stage. Any layout breaks, mobile responsiveness flaws, or caching errors are immediately detected and remediated on the staging link prior to approval.

### 4.2 Production Deployment
Once the Pull Request satisfies all architectural audits and receives explicit Unit approval, the branch is merged. Vercel automatically detects the merge event and instantaneously deploys the final compilation to the production-ready live site, securing the changes via robust SSL/TLS encryption.
